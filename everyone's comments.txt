Code Review Findings

✅ 1. Remove debug logging from production code

get_performance_reviews() includes a print("hey") statement that will pollute server logs on every list request.

File: src/api/routes/performance_reviews.py
Line: 28

✅ 2. DELETE performance review does not verify the row existed

delete_performance_row() always returns HTTP 204, even when review_id does not exist.

Manual testing showed that DELETE /performance_reviews/99999 returned 204 with an empty body. Callers cannot distinguish a successful delete from a no-op.

Recommendation: Verify that the row exists before deleting, or check the affected row count and return 404 Not Found when appropriate.

File: src/api/routes/performance_reviews.py
Lines: 118-127

✅ 3. Employee delete surfaces raw DB errors as HTTP 500

Deleting an employee who has comments or performance reviews returns 500 Internal Server Error instead of a structured 409 Conflict, or a clearly defined cascading delete policy.

This was tested on employee ID 9 after creating related records.

Recommendation: Catch foreign-key constraint errors and return a structured 409 Conflict response, or implement a deliberate cascade strategy.

File: src/api/routes/employees.py
Lines: 143-173

✅ 4. Employee create does not validate foreign keys before INSERT

POST /employees/ with company_id: 99999 returns 500 Internal Server Error instead of 404 or 422 with a clear message.

Recommendation: Validate foreign-key references before inserting, similar to how comments already validate employeeId and authorId.

File: src/api/routes/employees.py
Lines: 102-127

✅ 5. Inconsistent HTTP status codes for resource creation

POST /companies correctly returns 201 Created, but POST /employees/ returns 200 OK.

Recommendation: Align create endpoints on 201 Created for consistency and REST expectations.

Files: src/api/routes/companies.py, src/api/routes/employees.py

✅ 6. Typo in route handler name

get_tag() in titles.py should be renamed to get_title() for readability and searchability.

File: src/api/routes/titles.py
Line: 18

✅ 7. Inconsistent delete URL for titles

Titles are listed at /titles/ but deleted at /title/{title_id}/.

The singular path and trailing slash are inconsistent with the plural convention used elsewhere and may be easy to misuse.

Recommendation: Use a plural route such as DELETE /titles/{title_id}.

File: src/api/routes/titles.py
Line: 82

✅ 8. Unused imports and dead code in performance reviews

Field and field_validator are imported from Pydantic but never used.

Recommendation: Either add validation, such as rating ranges or date ordering, or remove the unused imports.

File: src/api/routes/performance_reviews.py
Line: 2

✅ 9. API_KEY configured in deployment but never enforced

render.yaml defines API_KEY, but neither server.py nor any route checks it.

Recommendation: Either implement authentication middleware or remove the unused environment variable to avoid a false sense of security.

Files: render.yaml, src/api/server.py

✅ 10. Default database credentials in source

db.py falls back to postgresql+psycopg://myuser:mypassword@localhost/pr_database when POSTGRES_URI is unset.

Recommendation: Fail fast in non-local environments so default credentials are never committed, assumed, or accidentally used in production.

File: src/api/db.py
Lines: 7-10

💛 11. No shared data-access layer; duplicated patterns
**We are not doing this because it is unnecessary for the current scope.**

Each route opens db.engine.begin() and runs raw SQL inline.

Recommendation: Consider adding a thin repository module or SQLAlchemy models to reduce duplication, centralize error handling, and make foreign-key validation consistent across resources.

Files: all files under src/api/routes/

✅ 12. get_employee returns path param as id without re-reading from row

The handler uses id=employee_id from the URL rather than id=employee.id from the query result.

Today these match, but if the query ever changes, this could mask bugs.

Recommendation: Prefer using id=employee.id.

File: src/api/routes/employees.py
Line: 55

✅ 13. DELETE endpoints return bodies where spec says no response

APISpec.md documents <No Response> for deletes on employees, performance reviews, and comments.

However, the implementation returns Employee or Title on some deletes and 204 No Content on others.

Recommendation: Standardize all delete endpoints on 204 No Content with no response body.

Files: employees.py, titles.py, comments.py, performance_reviews.py

❌ 14. No automated tests

The repo has manual test markdown files but no pytest or similar automated test suite.

Recommendation: Add a small integration test module to catch regressions such as delete behavior, filtering bugs, and foreign-key validation errors.

✅ 15. Python version mismatch between pyproject.toml and Render

pyproject.toml requires Python >=3.14, while render.yaml pins Python 3.12.9.

This may cause local, development, and production environments to diverge.

Recommendation: Align the Python versions across configuration files.

Files: pyproject.toml, render.yaml

✅ 1. Several endpoints are missing explicit `response_model` declarations, especially in `companies.py` and `performance_reviews.py`. Adding response models would make the API documentation clearer and make responses more predictable.

✅ 2. `performance_reviews.py` imports `Field` and `field_validator`, but does not use them. 

✅ 3. `PerformanceReview` is used as the request body for creating a review, but it does not include an `id` while the returned object does. A separate response model should be created for returned reviews.

✅ 4. `companies.py` only defines a `NewCompany` model but does not define a full `Company` response model. This makes the company endpoint less consistent with resources like titles and employees.

❌ 5. Some SQL queries return `.mappings()` while others return row objects directly. The project should use one style consistently.

✅ 6. The title endpoints use list-style parameter passing for single inserts/deletes in some places.

```python
[{"tid": title_id}]
```

A plain dictionary would be clearer for single-row operations.

✅ 7. The employee insert query uses shortened parameter names like `cid`, `fn`, `ln`, `tid`, and `dep`. More descriptive names like `company_id`, `first_name`, and `department` would help improve readability.

✅ 8. Several route functions do not include docstrings explaining what they do. Adding short docstrings would improve the generated FastAPI docs.

✅ 9. The comments endpoint checks whether the author exists and whether the employee exists using two separate queries. This could be simplified or moved into a helper function to reduce repetition.

❌ 10. The comment formatting function manually maps database column names to API field names. This is useful, but the pattern is only used for comments. Similar formatting helpers should be used consistently across resources.

✅ 11. `get_titles()` says “Retrieves all titles,” but `get_employees()` also says “Retrieves all employees” even though it only retrieves employees for one company. The docstring should describe the actual behavior.

✅ 12. The code repeats similar `HTTPException(status_code=404, detail=...)` patterns across resources. A helper function could reduce repetition and keep error messages consistent.

✅ 13. Several endpoints use trailing slashes inconsistently in their route decorators. This can cause small but annoying client-side issues

❌ 14. Some endpoints return database values directly as dictionaries, while others manually construct Pydantic models. Project should choose one pattern and apply it consistently.
